This document will guide you through the complete process of deploying the CMS on a Linux server (e.g., Debian/Ubuntu) from scratch to a functional website running under your own domain.\n\n## 1. Prerequisites\n\nBefore you begin, ensure that you have the following tools installed on your server:\n- `git`\n- `python3` and `python3-venv`\n- `nginx`\n- `certbot` (recommended for SSL)\n\nYou can install them easily using the package manager:\n```bash\napt update\napt install git python3 python3-venv nginx\n```\n\n## 2. Environment Preparation\n\nIt is recommended to run the application under an unprivileged user, such as `www-data`.\n\n#### Step 1: Clone the Repository\nClone or copy the project into the target directory. We will use `/var/www/DOMAIN` as an example.\n\n```bash\n# Create the directory for the project:\nmkdir -p /var/www/DOMAIN\nmkdir -p /var/www/DOMAIN/run\n\n# Clone the project:\ngit clone <REPOZITORY> /var/www/DOMAIN\n\n# Set the correct permissions:\nchown www-data:www-data -R /var/www/DOMAIN\n```\n\n#### Step 2: Create a Virtual Environment\nThe application will run in an isolated Python environment.\n\n```bash\n# Switch to the project directory\ncd /var/www/DOMAIN\n\n# Create the venv (as the www-data user)\nsudo -u www-data python3 -m venv venv\n```\n\n#### Step 3: Install Dependencies\nInstall all necessary Python libraries.\n\n```bash\n# Run pip from our venv\nsudo -u www-data ./venv/bin/pip install -r requirements.txt\n```\n\n## 3. Systemd Configuration\n\nFor reliable background operation, we will use `systemd` with socket-based activation. `systemd` will listen on a socket and start the application only upon the first incoming request. We will use Uvicorn as the Python wrapper.\n\n#### Step 1: Create the Socket File\nCreate the file `/etc/systemd/system/uvicorn-DOMAIN.socket`:\n```\n# /etc/systemd/system/uvicorn-DOMAIN.socket\n[Unit]\nDescription=Uvicorn socket for CMS\n\n[Socket]\nListenStream=/var/www/DOMAIN/run/uvicorn-DOMAIN.sock\nSocketUser=www-data\nSocketGroup=www-data\nSocketMode=0660\n\n[Install]\nWantedBy=sockets.target\n```\n\n!!! Ensure that the directory `/var/www/DOMAIN/run/` exists and has the correct permissions so that the `www-data` user can write to it.\n\n#### Step 2: Create the Service File\nCreate the file `/etc/systemd/system/uvicorn-DOMAIN.service`:\n```\n# /etc/systemd/system/uvicorn-DOMAIN.service\n[Unit]\nDescription=Uvicorn service for CMS\nRequires=uvicorn-DOMAIN.socket\nAfter=network.target\n\n[Service]\nUser=www-data\nGroup=www-data\nWorkingDirectory=/var/www/DOMAIN\nEnvironment=\"PYTHONPATH=/var/www/DOMAIN/venv/lib/python3.11/site-packages\"\nExecStart=/var/www/DOMAIN/venv/bin/uvicorn main:app --workers 3 --uds /var/www/DOMAIN/run/uvicorn-DOMAIN.sock --proxy-headers\nRestart=on-failure\n\n[Install]\nWantedBy=multi-user.target\n```\n\n!! Do not forget to adjust the path in `PYTHONPATH` according to your Python version (e.g., `python3.11`, `python3.13`, etc.).\n\n#### Step 3: Start and Enable Services\n```bash\n# Reload the systemd configuration\nsystemctl daemon-reload\n\n# Enable and immediately start the socket\nsystemctl enable --now uvicorn-DOMAIN.socket\n\n# Verify that the socket is listening\nsystemctl status uvicorn-DOMAIN.socket\n```\n\n## 4. Nginx Configuration\n\nNginx will serve as a reverse proxy‚Äîit will accept requests from the internet and forward them to our application running on the socket. It will also handle the efficient serving of static files (CSS, JS, images).\n\nCreate the file `/etc/nginx/sites-available/DOMAIN.conf` (replace DOMAIN with your domain):\n\n```nginx\nserver {\n    listen 80;\n    listen [::]:80;\n\n    server_name DOMAIN;\n    root /var/www/DOMAIN;\n\n    location ^~ /.well-known/acme-challenge/ {\n       alias /var/www/DOMAIN/.well-known/acme-challenge/;\n    }\n\n    # Redirect to HTTPS and handler for Certbot\n    location / {\n        return 302 https://$host$request_uri;\n    }\n}\n\nserver {\n    listen 443 ssl;\n    listen [::]:443 ssl;\n\n    http2 on; We recommend running `certbot --nginx` for the first time to automatically obtain and set up SSL certificates.\n\n## 5. Final Check\n\nIf you have followed the steps correctly, your CMS should now be accessible on your domain.\n\nYou can monitor any application errors using:\n```bash\njournalctl -u uvicorn-DOMAIN.service -f\n```",

    "markdown_content": "! This CMS includes a robust system for user management and restricting content access. Everything is controlled using simple `YAML` files.\n\n### Access Control\n\nThe key to securing pages is the `access` YAML block in the header of every `default.md` file. It allows you to define who can view the page.\n\n#### Common Examples:\n\n**1. Require Any Login:**\nThis is the most frequent case. Anyone who is logged in can access the page, regardless of their role.\n```yaml\naccess:\n    site.login: true\n```\n\n**2. Require Administrator Rights:**\nOnly users who have the `admin.login: true` permission set in their profile can access the page.\n```yaml\naccess:\n    admin.login: true\n```\n\n**3. Access Only for a Specific User:**\nYou can also create entirely custom rules. If you want to restrict access only to the user `zden2k`, you first define a unique right in their file `user/accounts/zden2k.yaml`:\n```yaml\n# File: user/accounts/zden2k.yaml\naccess:\n  site:\n    login: true\n  zden2k:\n    login: true # Unique right\n```\nAnd then you require this specific permission on the page:\n```yaml\n# File: default.md of the given page\naccess:\n    zden2k.login: true\n```\n\n---\n### Visibility in Menu vs. Access\n\n!!!! It is important to distinguish between visibility and access!\n\n*   **`visible: false`**\n    This `YAML` key **only hides the item from the main menu**.
